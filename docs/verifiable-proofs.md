# Verifiable Proofs: Intel TDX, the Nonce Flow, and livy-tee

This document explains the security model behind TDX attestation, how Intel's official
Go CLI implements the verifier nonce to protect against relay attacks, and how livy-tee
achieves the same security guarantees while also producing *self-verifiable proofs* —
proofs that can be fully reconstructed and re-checked by any third party long after the
original attestation session is gone.

---

## 1. The Intel TDX security model

Intel Trust Domain Extensions (TDX) is a hardware confidential computing technology that
lets you run a *Trust Domain* (TD) — a virtual machine whose memory is encrypted and
integrity-protected by the CPU. The hypervisor cannot read or tamper with the TD's state.

The core mechanism for proving to the outside world that specific code ran inside a
genuine TD is the **DCAP quote** — a ~8 KB cryptographic attestation document signed by
the CPU's hardware root-of-trust:

```
┌──────────────────────────────────────────────────────────────────────────────┐
│  TDX DCAP Quote (8 000 bytes, variable)                                      │
│                                                                              │
│  [0..8]      ─ Header (magic, version, attestation key type)                 │
│  [8..568]    ─ TD Report Body                                                │
│  [8..56]     ─ MRTD      — 48-byte hash of the loaded TD image (binary)      │
│  [56..120]   ─ RTMRs     — 4 × 48-byte runtime measurement registers         │
│  [568..632]  ─ REPORTDATA — 64 bytes chosen by the TD at quote time          │
│  [632..]     ─ Quoting Enclave signature + certificate chain                 │
└──────────────────────────────────────────────────────────────────────────────┘
```

The key property is **REPORTDATA**: the TD can write any 64 bytes here before requesting
a quote. Because the subsequent signature covers the entire TD Report Body, whatever is
placed in REPORTDATA is irrevocably bound to the TD's hardware identity (MRTD) in the
quote. This is the hook on which all higher-level provenance and binding schemes hang.

**MRTD** is particularly important: it is the hardware's own measurement of the binary
that was loaded into the TD at boot time. If two separate machines run the same binary
and produce quotes with the same MRTD, anyone can verify those quotes came from the same
code — without trusting the vendor.

## 2. The relay-attack problem and Intel's verifier nonce solution

### What is a relay attack?

Suppose you request a DCAP quote from a service claiming to run inside a TD. A malicious
intermediary could *intercept* your request, forward it to a different (or older) machine
that generates a valid quote, and relay that quote back to you. You would receive a
genuine Intel-signed document — but it would not correspond to the session you initiated.

This is feasible because:
- A valid DCAP quote has no expiry by itself.
- The REPORTDATA content is chosen by whoever requests the quote; an attacker controlling
  the quoting side can put anything there.
- Without a fresh, unpredictable challenge embedded in the quote, you cannot tell whether
  the quote was generated *now* and *for you*.

### Intel's solution: the ITA verifier nonce

Intel Trust Authority (ITA) counters relay attacks by issuing a *verifier nonce* — a
short-lived, server-signed challenge — before any attestation takes place.

The sequence defined by Intel is:

```
Client (TD side)                   Intel Trust Authority
     │                                     │
     │──── GET /appraisal/v2/nonce ───────►│
     │                                     │ ← generates VerifierNonce
     │◄─── { val, iat, signature } ────────│
     │                                     │
     │  Compute:                           │
     │  REPORTDATA = SHA-512(              │
     │    nonce.val ‖ nonce.iat ‖ data     │
     │  )                                  │
     │                                     │
     │  Request TDX quote with             │
     │  this REPORTDATA via configfs       │
     │                                     │
     │──── POST /appraisal/v2/attest* ────►│
     │     { quote, runtime_data, nonce }  │ ← verifies SHA-512 binding server-side
     │◄─── { token: <JWT> } ───────────────│
```

\* On Azure CVMs the attestation endpoint is `/appraisal/v2/attest/azure`.

The nonce fields:
- **`val`** — random bytes, base64-encoded, generated by ITA's HSM.
- **`iat`** — issued-at timestamp as bytes, base64-encoded.
- **`signature`** — ITA's HMAC over `val ‖ iat`, proving the nonce is genuine.

When ITA receives the attestation request it:
1. Re-computes `SHA-512(nonce.val ‖ nonce.iat ‖ runtime_data)`.
2. Extracts REPORTDATA bytes `[568..632]` from the raw quote.
3. Asserts they are equal.

If they match, ITA knows:
- The quote was generated *after* the nonce was issued (freshness).
- The quote was generated with knowledge of the specific nonce ITA provided (binding).
- An attacker cannot replay a pre-captured quote because it would contain the wrong nonce.

The signed JWT that ITA returns then carries the attested claims — MRTD, TCB status,
report data — as a credential anchored to Intel's PKI.

## 3. How the Intel Go CLI implements this

The Intel `trustauthority-client-for-go` CLI is the reference implementation. The
relevant flow lives in three files:

### `connector/connector.go` — nonce fetch

```go
// GET /appraisal/v2/nonce
func (c *trustAuthorityConnector) GetNonce(args GetNonceArgs) (*VerifierNonce, *RequestId, error) {
    // Issues an HTTPS GET to the ITA endpoint.
    // Returns a VerifierNonce with Val []byte, Iat []byte, Signature []byte.
}
```

### `tdx/tdx_adapter.go` — REPORTDATA construction

```go
// Collect collects TDX evidence given a nonce and optional user data.
func (a *TdxAdapter) CollectEvidence(nonce *verifier.VerifierNonce, userData []byte) (*verifier.Evidence, error) {

    // Step 1: build the 64-byte REPORTDATA
    reportData := make([]byte, 64)
    hash := sha512.New()
    hash.Write(nonce.Val)
    hash.Write(nonce.Iat)
    hash.Write(userData)          // up to 64 bytes of caller-supplied data
    copy(reportData, hash.Sum(nil))

    // Step 2: write reportData to TSM configfs and get the quote
    quote, err := a.tdxQuote(reportData)
    ...

    // Step 3: return Evidence containing raw quote
    return &verifier.Evidence{
        Type:        verifier.EvidenceType_TDX,
        Evidence:    quote,
        UserData:    userData,      // original bytes — sent as runtime_data
        VerifierNonce: nonce,
    }, nil
}
```

### `connector/connector.go` — attestation request body

```go
// The JSON body sent to POST /appraisal/v2/attest:
type TdxEvidence struct {
    Quote          []byte        `json:"quote"`           // base64url raw DCAP quote
    RuntimeData    []byte        `json:"runtime_data"`    // base64url original user data
    VerifierNonce  *VerifierNonce `json:"verifier_nonce"` // { val, iat, signature }
}
```

ITA verifies `SHA-512(nonce.val ‖ nonce.iat ‖ runtime_data) == REPORTDATA` server-side
and, if it passes, issues a signed JWT with claims including `tdx_mrtd` and
`tdx_report_data` (which is the SHA-512 hash, not the raw user data).

### The Intel CLI limitation: ephemeral nonces

The Intel CLI flow is correct and secure at attestation time. But it has an architectural
consequence: **the nonce is ephemeral**. Once the ITA session is over, the nonce is gone.

If you store only the JWT and want to re-verify it later, you face a problem:

```
JWT { tdx_report_data: "SHA-512(nonce‖data)" }
             │
             │  Want to verify: did this JWT cover my original input?
             │
             ▼
     You need: nonce.val, nonce.iat, original data
     Intel CLI never stored these → reconstruction impossible
```

The JWT's `tdx_report_data` field is the SHA-512 *hash* of the nonce and data combined.
Given only the JWT and the original data, you cannot reverse the SHA-512 or recover the
nonce, so you cannot confirm the JWT covers your specific data. The proof is
**cryptographically opaque** after the session ends.

This was confirmed by running both tools side-by-side on GCP TDX hardware:

```
Intel CLI JWT:
  tdx_mrtd:        "a1b2c3..."   ← same hardware measurement
  tcb_status:      "OutOfDate"   ← same firmware state
  tdx_report_data: "sha512hash"  ← opaque: SHA-512(nonce‖raw_bytes)
  (no nonce stored, no user_data stored in JWT)

livy-tee JWT:
  tdx_mrtd:        "a1b2c3..."   ← identical MRTD (same binary)
  tcb_status:      "OutOfDate"   ← identical TCB status
  tdx_report_data: "sha512hash"  ← same scheme: SHA-512(nonce‖runtime_data)
```

Both JWTs are valid and carry the same hardware claims. The difference is in what is
stored alongside the JWT for future verification.

## 4. How livy-tee achieves the same security (and more)

livy-tee adopts the Intel CLI's nonce flow exactly — same REPORTDATA construction and
request body format, with endpoint selection matching provider (`/attest` vs
`/attest/azure`). ITA's server-side verification passes for livy-tee quotes for the
same reason it passes for Intel CLI quotes.

The additional property livy-tee provides is **proof portability**: by storing the nonce,
runtime data, and portable evidence artifact in the `Attestation` struct, the local
binding chain can be reconstructed offline on non-Azure guests and the bundled evidence
can be reappraised later on any verifier that has ITA credentials.

For Azure CVMs, Intel Trust Authority validates Azure runtime JSON via the
`/appraisal/v2/attest/azure` path. The ITA token remains the source of truth for
TCB/MRTD claims in that provider flow.

### 4.1 What livy-tee stores in `Attestation`

```rust
pub struct Attestation {
    pub ita_token:          String,     // ITA-signed JWT
    pub jwks_url:           String,     // JWKS endpoint matching the ITA region
    pub mrtd:               String,     // hex MRTD (48 bytes)
    pub tcb_status:         String,     // "UpToDate" | "OutOfDate" | "Revoked"
    pub tcb_date:           Option<String>,
    pub evidence:           String,     // portable evidence artifact; Azure keeps runtime JSON here
    pub raw_quote:          String,     // base64 raw DCAP quote (~8 KB)
    pub runtime_data:       String,     // base64 of our 64-byte ReportData struct
    pub verifier_nonce_val: String,     // base64 nonce.val bytes
    pub verifier_nonce_iat: String,     // base64 nonce.iat bytes
    pub verifier_nonce_signature: String,
    pub report_data:        ReportData, // parsed structured payload
    pub public_values:      PublicValues,
}
```

### 4.2 The two-level binding chain

livy-tee implements a two-level cryptographic binding:

```
Level 1 — Payload binding (SHA-256, application layer):

  public_values buffer ──SHA-256──► payload_hash  [ReportData bytes 0..32]


Level 2 — ITA nonce binding (SHA-512, ITA protocol layer):

  nonce.val  ─┐
              ├─SHA-512──► REPORTDATA [quote bytes 568..632]
  nonce.iat  ─┤
              │
  runtime_data (= 64-byte ReportData struct serialized) ─┘
```

The ReportData struct's wire layout is:

```
Bytes   Size  Field         Content
─────   ────  ─────         ───────
00..32   32   payload_hash  SHA-256(public_values buffer)
32..40    8   build_id      First 8 bytes of SHA-256(server binary) — build fingerprint
40..44    4   version_code  Schema version (currently 1)
44..48    4   build_number  CI build counter
48..56    8   nonce         Monotonic request counter (application-level replay protection)
56..64    8   reserved      Zero-filled
```

### 4.3 Binding reconstruction by any third party

Given an `Attestation` and the expected public values buffer, any party can reconstruct
the local binding chain with nothing but standard cryptography libraries on non-Azure
guests:

```
Step 1 — Recompute payload_hash from expected public values:
  expected_payload = SHA-256(public_values.buffer)

Step 2 — Decode runtime_data:
  rd_bytes = base64_decode(attestation.runtime_data)
  rd = ReportData::from_bytes(rd_bytes)    // 64-byte struct

Step 3 — Assert payload binding:
  assert rd.payload_hash == expected_payload

Step 4 — Reconstruct REPORTDATA:
  nonce_val = base64_decode(attestation.verifier_nonce_val)
  nonce_iat = base64_decode(attestation.verifier_nonce_iat)
  expected_rd = SHA-512(nonce_val ‖ nonce_iat ‖ rd_bytes)

Step 5 — Extract and verify REPORTDATA in the raw quote:
  quote_bytes = base64_decode(attestation.raw_quote)
  quote_rd    = quote_bytes[568..632]
  assert quote_rd == expected_rd

Step 6 — Verify the ITA JWT signature and expiry:
  fetch JWKS from attestation.jwks_url
  verify attestation.ita_token against the JWKS

Step 7 — Assert TCB and hardware identity policy:
  token.tcb_status == "UpToDate"  // default livy-tee policy
  rd.build_id == first_8_bytes(SHA-256(known_binary))
  attestation.mrtd == expected_mrtd_for_this_binary
```

Steps 1–5 require no network, no TDX hardware, and no Livy infrastructure. Any
developer, auditor, or end user can perform them with only:
- The `Attestation` fields (a few KB of data).
- The expected public values buffer.
- A SHA-256 and SHA-512 implementation (available in every language).

This is implemented in `livy_tee::verify_quote` and tested in
`tests/tdx_integration.rs::external_verifier_reconstructs_report_data_from_raw_inputs`.
`Attestation::verify()` performs the signed-token, TCB policy, and public-value
binding checks. `Attestation::verify_fresh()` performs the stricter path that
reappraises `attestation.evidence` with ITA and authenticates the bundled
evidence artifact itself.

For Azure CVMs, step 5 is not a provider-portable local check. The authoritative
path is:

```
Step A — Parse the portable evidence artifact:
  evidence = Evidence::from_transport_string(attestation.evidence)

Step B — Submit that evidence, runtime_data, and verifier nonce back to ITA:
  appraise_evidence_unauthenticated(evidence, runtime_data, verifier_nonce)

Step C — Assert the fresh ITA appraisal matches the attestation's public fields:
  fresh.mrtd == attestation.mrtd
  fresh.tcb_status == attestation.tcb_status
  fresh.tcb_date == attestation.tcb_date
  base64(evidence.raw_quote) == attestation.raw_quote
```

### 4.4 What livy-tee adds beyond the Intel CLI

| Property                                | Intel CLI | livy-tee |
|-----------------------------------------|-----------|----------|
| TSM configfs quote generation           | ✅        | ✅       |
| ITA v2 nonce anti-relay protection      | ✅        | ✅       |
| ITA server-side SHA-512 verification    | ✅        | ✅       |
| Valid ITA JWT with MRTD + TCB claims    | ✅        | ✅       |
| Stored runtime_data + nonce             | ❌        | ✅       |
| Structured public-value payload binding | ❌        | ✅       |
| Offline non-Azure local binding reconstruction | ❌ | ✅ |
| Fresh ITA reappraisal of bundled evidence     | ❌ | ✅ |
| Application-level replay protection     | ❌        | ✅       |
| Build identity in payload (build_id)    | ❌        | ✅       |

The Intel CLI produces a correct attestation but treats proofs as opaque credentials —
a "yes/no" from Intel at a single point in time. livy-tee treats proofs as portable,
self-contained records that carry all material needed to re-derive every hash in the
chain from the original data.

## 5. The two nonces in livy-tee

livy-tee uses two distinct nonces that operate at different layers of the system:

### 5.1 ITA verifier nonce (anti-relay)

- **Who issues it:** Intel Trust Authority server.
- **Where it lives:** `Attestation.verifier_nonce_val` + `Attestation.verifier_nonce_iat`.
- **What it protects against:** Relay attacks — an attacker cannot present a pre-captured
  quote as fresh because ITA will not accept a quote whose REPORTDATA does not match the
  SHA-512 of the nonce it just issued.
- **Scope:** One per attestation session (per `commit()` call).
- **Expiry:** ITA rejects nonces after a short window (typically seconds).
- **Stored in Attestation:** Yes — necessary for offline reconstruction of the SHA-512 chain.

### 5.2 Application nonce (anti-replay)

- **Who issues it:** The application (your code), passed via `.nonce(counter)`.
- **Where it lives:** `ReportData.nonce` (bytes 48..56 of the 64-byte struct).
- **What it protects against:** Replay attacks — an attacker cannot reuse a valid proof
  for a different request because the proof carries an embedded request sequence number.
  The server stores the nonce alongside the record; a verifier confirms the nonce in the
  quote matches the stored value.
- **Scope:** Per request / per ingestion. Use a persistent monotonic counter.
- **Stored in Attestation:** Yes — it is part of the ReportData struct in `runtime_data`.

```
ITA verifier nonce:  "This quote was generated NOW, in response to THIS nonce challenge"
Application nonce:   "This quote was generated for REQUEST #42, not a replay of #17"
```

Both nonces are embedded in the hardware-signed quote. Neither can be forged or stripped
without invalidating the signature.

## 6. Summary

Intel TDX's REPORTDATA mechanism provides a 64-byte slot in every hardware-signed quote.
Intel's nonce protocol protects that slot from relay attacks by requiring the TD to prove
it received a fresh ITA challenge before the quote was generated.

livy-tee implements this protocol fully (same endpoints, same SHA-512 construction, same
request body format as the Intel Go CLI) and additionally:

1. **Structures the `runtime_data`** as a typed 64-byte `ReportData` payload that embeds
   public-value binding, binary identity, and a replay-protection counter.
2. **Stores all reconstruction materials** (`runtime_data`, `verifier_nonce_val`,
   `verifier_nonce_iat`, `raw_quote`) in the `Attestation` struct.
3. **Provides `verify_quote`** — a pure-software, offline function that any external
   party can call to re-derive the complete hash chain from expected public values
   with no TDX hardware, no network, and no Livy infrastructure.

The result is that livy-tee proofs are not just valid Intel attestations — they are
permanently self-verifiable records that carry their own verification key material.
